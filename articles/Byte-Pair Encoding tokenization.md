# Byte-Level BPE tokenizer とは何か

この記事では テキストをトークン列に変換する手法である Byte-Level BPE (Byte-Level Byte Pair Encoding) という tokenizer について解説します。

この記事を書くにあたって、まず Hugging Face の LLM Course にある以下の記事を参考にさせていただきました。

[Byte-Pair Encoding tokenization](https://huggingface.co/learn/llm-course/chapter6/5) 


## BPE とは何か

BPE は Byte Pair Encoding の略です。BPE の考え方を一言でいうと、「よく一緒に出てくる文字のペアを、どんどん一つの token としてまとめていく」というものです。

例えば、最初は単語を文字単位にバラバラにします。「hug」という単語であれば、最初は「h」「u」「g」のように分けて考えます。ここで大量の文章を見ていくと、「u」と「g」が続けて出てくることが多い、ということが分かるかもしれません。その場合、BPE は「u」と「g」をまとめて「ug」という新しい token にします。次にまた文章全体を見て、今度は「h」と「ug」がよく一緒に出てくるなら、それをまとめて「hug」という token にします。

このように、最初は小さな文字単位から始めて、頻繁に出てくる組み合わせを少しずつ大きな token にしていくのが BPE の基本です。

## BPE の学習の流れ

BPE の学習では、まずコーパスと呼ばれる大量の文章データを用意します。そして、その文章を単語のような単位に分けます。これを pre-tokenization と呼びます。日本語だと少し分かりにくいですが、tokenizer 本体に入る前の前処理のようなものです。

次に、それぞれの単語をさらに文字単位に分解します。この時点では、語彙はかなり小さいです。英語であればアルファベットや記号などが最初の語彙になります。

そのあと、コーパス全体を見て、隣り合って出てくる token のペアを数えます。例えば「u」と「g」が一番多く隣り合って出てくるなら、それを merge して「ug」という新しい token を作ります。そしてまた同じことを繰り返します。

次は「h」と「ug」が多いかもしれませんし、「t」と「h」が多いかもしれません。とにかくその時点で一番よく出てくる隣り合った組み合わせを一つにまとめます。これを決められた語彙サイズになるまで繰り返します。

例えば語彙サイズを 50,000 にすると決めたら、最初の文字や記号に加えて、頻繁に出てくる subword がどんどん語彙に追加されていきます。

ここで重要なのは、BPE は最初から「単語」を覚えるわけではないということです。よく出てくる単語は最終的に一つの token になるかもしれません。一方で、あまり出てこない単語は、複数の subword に分けられます。

この性質がかなり便利です。完全な単語単位の tokenizer だと、知らない単語が出てきたときに unknown token になってしまいます。しかし BPE では、知らない単語でも、すでに知っている小さな部品に分けることができます。例えば「tokenization」という単語を完全な一単語として知らなくても、「token」「ization」のような部品を知っていれば、それらの組み合わせとして扱えます。

このおかげで、BPE は単語単位よりも柔軟で、文字単位よりも効率が良い tokenizer になります。

## tokenization のときは何をしているのか

学習が終わると、BPE tokenizer には merge rule が残ります。merge rule とは、「このペアが出てきたら、この token にまとめる」というルールのことです。

新しい文章を tokenize するときは、学習時と同じように、まず文章を前処理して、文字単位に分けます。そのあと、学習で得られた merge rule を順番に適用していきます。

例えば、学習時に「u」と「g」を「ug」にまとめるルールを覚えていて、さらに「h」と「ug」を「hug」にまとめるルールも覚えていたとします。この場合、「hug」という文字列が入力されると、まず「u」と「g」がまとまり、次に「h」と「ug」がまとまって、最終的に「hug」という token になります。

ここで大事なのは、tokenizer はその場で自由に考えて分割しているわけではないということです。あくまで、学習時に決まった merge rule を順番に適用しているだけです。そのため、同じ tokenizer を使えば、同じ入力は基本的に同じ token 列になります。これはモデルにとって非常に重要です。

## 普通の BPE の問題点

ここまで見ると、BPE はかなりうまくできているように見えます。ただし、普通の BPE には一つ大きな問題があります。それは、最初の語彙に存在しない文字が出てきたときです。

例えば、英語の文章だけで tokenizer を学習したとします。この場合、最初の語彙には英語のアルファベットやよく使われる記号しか入っていないかもしれません。そこに絵文字や、日本語の文字、特殊な記号などが入力されると、tokenizer はそれをうまく扱えない可能性があります。

このとき unknown token に変換されてしまうと、元の文字情報が失われます。これは意外と大きな問題です。

LLM は世界中の言語、絵文字、URL、コード、記号など、かなり雑多なテキストを扱う必要があります。そのたびに unknown token が出てしまうと、モデルは入力の細かい情報を失ってしまいます。そこで出てくるのが Byte-Level BPE です。

## Byte-Level BPE とは何か

Byte-Level BPE は、文字ではなく byte を最小単位として扱う BPE です。

普通の BPE では、最初の語彙は文字から始まります。英語であれば「a」「b」「c」のような文字、日本語であれば「あ」「い」「う」のような文字を考えることになります。一方で Byte-Level BPE では、テキストをまず byte の列として見ます。

コンピュータ上のテキストは、最終的には byte で表現されています。UTF-8 であれば、英語の文字は 1 byte で表されることが多く、日本語や絵文字は複数 byte で表されます。つまり、どんな文字であっても、byte の列としては表現できます。

Byte-Level BPE は、この性質を利用します。最初の語彙を 256 種類の byte にしておけば、理論上、どんな文字でも byte の組み合わせとして表現できます。これにより、普通の BPE で問題になっていた unknown token をかなり避けることができます。

Hugging Face の記事でも、GPT-2 や RoBERTa の tokenizer は Unicode 文字そのものではなく byte として見ることで、基本語彙を小さく保ちながら、ほとんどあらゆる文字を扱えるようにしていると説明されています。

## なぜ byte にすると嬉しいのか

Byte-Level BPE の一番大きなメリットは、未知の文字に強いことです。

普通の BPE では、学習時に見たことがない文字が出てくると困ります。しかし Byte-Level BPE では、どんな文字も最終的には byte に分解できます。例えば絵文字も、日本語も、アクセント付きの文字も、コードに出てくる特殊記号も、byte の列として表現できます。

そのため、tokenizer が「この文字は知らないので処理できません」となりにくいです。これは LLM にとってかなり重要です。

なぜなら、現実のテキストは綺麗な英語だけではないからです。SNS の文章には絵文字がありますし、プログラミングのコードには記号がたくさんあります。Web ページには URL や HTML のような文字列も出てきます。Byte-Level BPE は、そのような雑多な入力を一つの仕組みで扱いやすくしてくれます。

## ただし、byte だけで終わりではない

ここで誤解しやすいのは、Byte-Level BPE は「全部を byte 単位のまま扱う tokenizer」ではないということです。

もし本当にすべてを byte 単位のまま扱うと、文章がかなり長い token 列になってしまいます。そこで BPE の merge が効いてきます。

最初は byte から始めますが、学習によって頻繁に出てくる byte の組み合わせをどんどん merge していきます。英語でよく出る単語や subword は、最終的に一つの token になることがあります。日本語や絵文字のように複数 byte で表現される文字も、よく出てくる組み合わせであれば、より大きな token として扱われる可能性があります。

つまり Byte-Level BPE は、「byte から始めることで未知の文字に強くしつつ、BPE の merge によってよく出る表現は効率よくまとめる」仕組みだと言えます。

## 空白の扱いも重要

BPE tokenizer を見ていると、たまに「Ġ」のような見慣れない記号が出てくることがあります。これは GPT-2 系の tokenizer で、空白を表すために使われることがある記号です。

例えば「is」と「 is」は、人間から見ると同じ単語に見えますが、モデルにとっては前に空白があるかどうかも重要な情報です。英語では、単語の前に空白があることで単語の区切りが分かります。

そのため、Byte-Level BPE では空白も単なる区切りとして捨てるのではなく、token の一部として扱うことがあります。これにより、「文頭に出てくる単語」と「空白の後に出てくる単語」を区別できます。

最初は少し気持ち悪く見えますが、tokenizer の出力をよく見ると、この空白の扱いがかなり重要だと分かります。

## まとめ

今回、Hugging Face の記事を読みながら、BPE と Byte-Level BPE の仕組みを整理しました。

BPE は、最初に文字単位から始めて、よく出てくる隣り合った token のペアを merge していく tokenizer です。これにより、よく出てくる単語や subword は短い token 列で表現でき、知らない単語でも小さな部品に分けて扱うことができます。

一方で、普通の BPE では、最初の語彙に存在しない文字が出てきたときに unknown token になってしまう問題があります。Byte-Level BPE は、この問題を byte を最小単位にすることで解決します。

最初の語彙を 256 種類の byte にしておけば、英語、日本語、絵文字、記号など、ほとんどあらゆる文字を byte の組み合わせとして扱えます。そしてその上で BPE の merge を行うため、未知の文字に強く、かつ頻出パターンは効率よく token 化できます。

tokenizer は一見地味な部分ですが、LLM にとってはかなり重要な入り口です。モデルがどれだけ賢くても、入力の文章をうまく token に変換できなければ、その情報を正しく扱うことはできません。

Transformer の勉強をしていると、つい attention や loss function のようなモデル本体に目が行きがちですが、tokenizer も同じくらい丁寧に理解しておく必要があると感じました。
